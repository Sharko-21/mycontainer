# MyContainer Commander

Данный код представляет собой самописный аналог Docker на Go под Linux. Он отвечает за архитектуру жизненного цикла контейнеров, сохранение состояния в локальную СУБД SQLite, ограничение ресурсов через **Cgroups v2**, изоляцию окружения с помощью **Linux Namespaces**, подмену корневой файловой системы через `pivot_root` (с использованием OverlayFS) и запуск целевого процесса внутри контейнера.

## 🚀 Как это работает (Архитектура запуска)

Проект использует многошаговый жизненный цикл с сохранением метаданных контейнеров в базу данных и паттерн **Fork-Exec** (через повторный вызов `/proc/self/exe`) для безопасной инициализации изолированной среды.

Процесс работы разделен на следующие фазы:

1. **Команда `create`:**
* Генерирует уникальный UUID контейнера (ID).
* Создает изолированную структуру директорий для OverlayFS: `upper` (слой записи), `work` (служебная папка ядра) и `merged` (точка слияния слоев).
* Регистрирует контейнер в локальной SQLite базе данных со статусом `created` и сохраняет все переданные лимиты ресурсов.


2. **Команда `start` (Родительский хост-процесс):**
* Извлекает метаданные контейнера из базы данных.
* Монтирует файловую систему `overlay`, объединяя базовый образ (`lowerdir`), созданные на этапе `create` слои записи (`upperdir`) и рабочую директорию (`workdir`) в общую точку `merged`.
* Запускает **самого себя** (`/proc/self/exe`) с внутренней подкомандой `child`, передавая ему ID контейнера.
* Настраивает **Linux Namespaces** через `SysProcAttr.Cloneflags` (`CLONE_NEWUTS`, `CLONE_NEWPID`, `CLONE_NEWNS`), изолируя имя хоста, таблицу процессов и пространство монтирования.
* Создает иерархию cgroups на хосте, задает жесткие лимиты ресурсов и добавляет PID созданного процесса в `cgroup.procs`.
* Меняет статус контейнера в БД на `running` и ожидает его завершения, после чего обновляет статус на `completed`/`error` и чисто удаляет директорию cgroup на хосте.


3. **Команда `child` (Внутренний процесс контейнера):**
* Вызывается неявно через `/proc/self/exe child <container_id>`.
* Изменяет сетевое имя хоста (`hostname`) на `my-container`.
* Вызывает `mountRootFS()`, которая делает корень приватным (`MS_PRIVATE`), делает `pivot_root`, монтирует изолированные `/proc` и `/sys`.
* Выполняет `syscall.Unshare(syscall.CLONE_NEWCGROUP)` для изоляции cgroup-пространства, после чего монтирует чистый `cgroup2` в `/sys/fs/cgroup`.
* С помощью `exec.Command` подменяет текущий процесс целевой командой пользователя (например, `/bin/sh`) с очищенными и безопасными переменными окружения (`PATH`, `TERM`).

## 🛠 Ключевые фичи и системные вызовы

* **Полноценный State Management:** Все контейнеры, их PID, статусы (`created`, `running`, `completed`, `error`) и конфигурации ресурсов персистентно хранятся в SQLite (`/var/lib/mycontainer/state.db`) с включенным режимом логирования WAL для бесконфликтной параллельной работы.
* **Изоляция ФС (`pivot_root` + OverlayFS):** Вместо устаревшего `chroot` используется связка OverlayFS и `syscall.PivotRoot`. Старый корень хоста перемещается во временную директорию контейнера `.put_old`, лениво размонтируется через `MNT_DETACH` и удаляется. Контейнер полностью отрезан от хост-системы.
* **Собственные `/proc` и `/sys`:** Внутри контейнера заново монтируются `procfs` и `sysfs`. Команда `ps` внутри контейнера видит исключительно процессы внутри контейнера (целевой процесс получит `PID 1`), а подсистема cgroups изолирована.
* **Управление ресурсами через Cgroups v2:** Ограничение ресурсов пишется в современную иерархию `/sys/fs/cgroup/system.slice/mycontainer-<id>`.
* **Защита от Fork-Bomb:** В коде жестко зашит лимит `pids.max = 100`, что не позволит процессам внутри контейнера заблокировать хост-машину бесконечным размножением процессов.

## 📋 Поддерживаемые команды и флаги

Утилита поддерживает CLI-интерфейс, разбитый на явные подкоманды. Флаги конфигурации ресурсов передаются на этапе **создания** контейнера.

### Доступные команды

* `create` — Подготовка структуры OverlayFS и запись конфигурации контейнера в БД.
* `start <container_id>` — Запуск ранее созданного контейнера.
* `list` — Вывод таблицы активных контейнеров. Поддерживает флаг `--all` для детального вертикального вывода всех параметров метаданных (включая пути OverlayFS и точные лимиты).
* `delete <container_id>` — Удаление метаданных контейнера из базы данных и очистка его физических директорий записи (`upper`, `work`, `merged`). Удаление запущенного контейнера заблокировано.

### Флаги конфигурации (применяются к командам)

| Флаг | Значение по умолчанию | Описание | Пример использования |
| --- | --- | --- | --- |
| `--rootfs` | *Обязательный* | Путь к распакованному образу ОС (слой `lowerdir`) | `--rootfs /tmp/alpine` |
| `--memory-limit` | `536870912` (512MB) | Максимальный лимит памяти в байтах (`memory.max`) | `--memory-limit 268435456` |
| `--memory-request` | `0` | Гарантированный минимум памяти в байтах (`memory.low`) | `--memory-request 128000000` |
| `--cpu-request` | `100` | Доля CPU (weights) в cgroups v2 (`cpu.weight`) | `--cpu-request 200` |
| `--cpu-limit` | `-1` (`max`) | Ограничение CPU в формате `[лимит_в_мкс] [период_в_мкс]` | `--cpu-limit 50000 100000` *(50% от 1 ядра)* |
| `--debug` | `false` | Переводит логгер в режим `slog.LevelDebug` | `--debug` |

> **Примечание по `--cpu-limit`:** Принимает два аргумента. Значение `max 100000` (или дефолтное `-1`) отключает лимиты. Период по умолчанию — `100000` мкс (100мс).

## 💻 Пример использования

### 1. Подготовка базового RootFS (Lower)

Скачаем и распакуем минимальный образ Alpine Linux:

```bash
mkdir -p /tmp/alpine && cd /tmp/alpine
wget http://dl-cdn.alpinelinux.org/alpine/v3.18/releases/x86_64/alpine-minirootfs-3.18.4-x86_64.tar.gz
tar -xvf alpine-minirootfs-3.18.4-x86_64.tar.gz
rm alpine-minirootfs-3.18.4-x86_64.tar.gz

```

### 2. Создание контейнера (Регистрация и OverlayFS)

Задаем лимит памяти в 256 МБ и квоту CPU в 50% от одного ядра. Флаг `--` отделяет параметры нашего рантайма от целевой команды:

```bash
sudo go run main.go create \
  --rootfs /tmp/alpine \
  --memory-limit 268435456 \
  --cpu-limit 50000 100000 \
  --debug \
  -- /bin/sh

```

*Вывод вернет ID созданного контейнера, например:* `Created container:  0008d22b0b4104e2c003d64078914dd1`

### 3. Просмотр списка контейнеров

```bash
# Компактный список
sudo go run main.go list

# Детальный просмотр путей монтирования и cgroup лимитов
sudo go run main.go list --all

```

### 4. Запуск контейнера

```bash
sudo go run main.go start 0008d22b0b4104e2c003d64078914dd1

```

Вы окажетесь внутри изолированного шелла Alpine Linux. Любые изменения файлов будут писаться в `/var/lib/mycontainer/containers/<id>/upper`, не изменяя базовый `/tmp/alpine`.

## ⚠️ Требования к окружению

1. **ОС:** Linux (ядро с поддержкой OverlayFS, namespaces и cgroups v2).
2. **Права:** Запуск строго от имени `root` (или через `sudo`) для возможности выполнения системных вызовов `mount`, `pivot_root`, `unshare` и управления каталогом `/sys/fs/cgroup`.
3. **Cgroups v2:** На хост-машине должна быть смонтирована единая иерархия cgroups v2 (стандарт для Ubuntu 22.04+, Debian 11+, Fedora и др.). Контроллеры `memory`, `cpu` и `pids` должны быть активны в `cgroup.subtree_control`.
