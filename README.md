# MyContainer Commander

Данный код представляет собой самописный аналог Docker на Go под Linux. Он отвечает за парсинг флагов, ограничение ресурсов через **Cgroups v2**, изоляцию окружения с помощью **Linux Namespaces**, подмену корневой файловой системы через `pivot_root` и запуск целевого процесса внутри контейнера.

## 🚀 Как это работает (Архитектура запуска)

Код использует классический паттерн **Fork-Exec** (похожий на то, как устроен `runc` в настоящем Docker) для безопасной инициализации контейнера. Запуск разделен на две стадии:

1. **Стадия `run` (Родительский процесс):**
* Вызывается пользователем (например, `go-docker run --rootfs ... /bin/sh`).
* Запускает **самого себя** (`/proc/self/exe`) с подкомандой `child`, передавая ему все необходимые настройки.
* Настраивает **Linux Namespaces** (`CLONE_NEWUTS`, `CLONE_NEWPID`, `CLONE_NEWNS`), создавая изолированные пространства для сети/хоста, процессов и монтирования.
* После завершения работы контейнера подчищает за собой cgroups на хосте.


2. **Стадия `child` (Процесс внутри контейнера):**
* Работает уже внутри изолированных Namespaces.
* Перемещает свой PID в **Cgroups v2** для лимитирования ресурсов.
* Меняет имя хоста на `my-container`.
* Вызывает `mountRootFS()`, чтобы полностью отрезать контейнер от файловой системы хоста.
* Через `exec.Command` заменяет себя на целевую команду (например, `/bin/sh`), которую запросил пользователь.

## 🛠 Ключевые фичи и системные вызовы

* **Изоляция ФС (`pivot_root`):** Вместо старого и небезопасного `chroot` используется связка `syscall.Mount(..., MS_BIND)` + `syscall.PivotRoot`. Старый корень хоста временно перемещается в `.put_old`, лениво размонтируется через `MNT_DETACH` и удаляется. Контейнер вообще не знает о существовании хост-системы.
* **Собственные `/proc` и `/sys`:** Внутри контейнера заново монтируются изолированные `procfs` и `sysfs`. Команда `ps` внутри контейнера покажет только процессы контейнера, а утилиты типа `ip link` — только его сеть.
* **Cgroups v2:** Ограничение ресурсов пишется в современную иерархию `/sys/fs/cgroup/my-container`.
* **Защита от Fork-Bomb:** В коде жестко зашит лимит `pids.max = 100`, что не позволит процессам внутри контейнера положить хост-машину бесконечным размножением.

## 📋 Поддерживаемые флаги

Флаги парсятся вручную перед выполнением команды. Регистр имеет значение.

| Флаг | Значение по умолчанию | Описание | Пример |
| --- | --- | --- | --- |
| `--rootfs` | *Обязательный* | Путь к папке с распакованным образом (rootfs) | `--rootfs /var/lib/alpine` |
| `--memory-limit` | `536870912` (512MB) | Максимальный лимит памяти в байтах (`memory.max`) | `--memory-limit 268435456` |
| `--memory-request` | `0` | Гарантированный минимум памяти в байтах (`memory.low`) | `--memory-request 128000000` |
| `--cpu-request` | `100` | Доля CPU (weights) в cgroups v2 (`cpu.weight`) | `--cpu-request 200` |
| `--cpu-limit` | `max` | Ограничение CPU в формате `[лимит] [период]` | `--cpu-limit 20000 100000` *(20% от 1 ядра)* |
| `--debug` | `false` | Включает вывод дебаг-логов логгера во время работы | `--debug` |

> **Примечание по `--cpu-limit**`: Если передано `max 100000`, контейнер может использовать CPU без ограничений. Период по умолчанию — `100000` мкс (100мс).

## 💻 Пример использования

### 1. Подготовка RootFS

Для работы контейнера нужна файловая система. Скачаем и распакуем, например, Alpine Linux:

```bash
mkdir -p /tmp/alpine && cd /tmp/alpine
wget http://dl-cdn.alpinelinux.org/alpine/v3.18/releases/x86_64/alpine-minirootfs-3.18.4-x86_64.tar.gz
tar -xvf alpine-minirootfs-3.18.4-x86_64.tar.gz
rm alpine-minirootfs-3.18.4-x86_64.tar.gz

```

### 2. Запуск контейнера

Поскольку создание Namespaces и Cgroups требует привилегий ядра, запускать собранный бинарник необходимо через `sudo`.

```bash
# Запуск контейнера с лимитом памяти 256MB и включенным дебагом
sudo ./mycontainer run \
  --rootfs /tmp/alpine \
  --memory-limit 268435456 \
  --debug \
  -- /bin/sh

```

*(Флаг `--` сообщает парсеру, что все последующие аргументы относятся к запускаемой внутри контейнера команде).*

## ⚠️ Требования к окружению

1. **ОС:** Только Linux (используются специфичные для Linux syscall'ы).
2. **Права:** Запуск от имени `root` (или с `sudo`), так как обычный пользователь не имеет прав на создание cgroups и монтирование файловых систем.
3. **Cgroups:** На хосте должны быть включены **Cgroups v2** (в современных дистрибутивах вроде Ubuntu 22.04+ / Debian 11+ они включены по умолчанию). Папка `/sys/fs/cgroup` должна быть доступна на запись.
